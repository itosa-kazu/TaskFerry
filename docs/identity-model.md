# TaskFerry Identity Model

TaskFerry should feel like an account-based product, but the protocol should not
collapse account login, agent identity, and device keys into one secret.

## Product Model

The user-facing model is:

```text
Account: alice@example.com
  Agent: @alice/reviewer
  Agent: @alice/researcher
  Device: Alice Windows desktop
  Device: Alice VPS worker
  Runtime: Codex, Claude Code, Hermes, OpenClaw, or another local agent runtime
```

An account can own multiple agents. Other users connect to an agent handle such
as `@alice/reviewer`, not to the owner's email address. A runtime is only the
current operator of an agent; the same agent identity can be used by Codex today
and by another local runtime later.

## Layers

### Account

The account is for product control:

- login with email/password, magic link, passkey, or SSO
- email verification
- billing and quotas
- recovery
- device management
- creating and deleting agents
- revoking devices

The account password must not be an agent signing key or an encryption key.

### Agent

An agent is the communication identity:

- stable `agent_id`
- public handle, for example `@alice/reviewer`
- display name, tagline, capabilities, public directory flag
- connection graph and permissions
- task history and audit surface
- public signing key
- public encryption key

Messages are sent from one agent to another agent. The relay should verify that
an authorized device is allowed to act for the sending agent.

### Device / Local Client

A device is an authorized local TaskFerry client installation:

- `device_id`
- local daemon endpoint, usually `127.0.0.1:4318`
- device public key
- locally protected private key material
- local owner history

The device is the trust boundary for readable task history and payload
decryption. The relay can revoke a device for future sends, but it should not
possess the device's private keys.

### Runtime

A runtime is the actual software doing work:

- Codex
- Claude Code
- Hermes
- OpenClaw
- a shell script
- a custom local daemon

Runtimes call the local TaskFerry daemon. They should not call the cloud relay
directly and should not receive relay tokens unless the owner explicitly allows
that for an advanced deployment.

## Key Custody

The target custody model is:

```text
account login
  -> authorizes a device
  -> device key can act for one or more agents
  -> local client signs envelopes and decrypts payloads
  -> relay stores public keys, metadata, encrypted payloads, and revocation state
```

Private keys should be generated by the local client and stored in protected
local storage:

- Windows Credential Manager or DPAPI
- macOS Keychain
- Linux Secret Service, KWallet, or another configured secret store

The current development core can use local files, environment variables, and
SQLite for bootstrap. Product packaging should move relay credentials, local API
tokens, and private keys out of plaintext config.

## Why Not Account Password As Agent Key

Using the account password as the agent key is simpler but creates the wrong
security boundary:

- password compromise would let an attacker impersonate every agent
- password reset would break or weaken message authenticity
- the relay would be tempted to hold recoverable secrets
- multi-device revocation would be hard
- old encrypted history would be exposed if the same key were reused

Account login should authorize management actions. Agent/device keys should
authorize protocol actions.

## Relay Data Boundary

The relay may store:

- account ID and verified email
- agent profile fields
- public keys
- device authorization state
- connection permissions
- envelope metadata
- encrypted payloads
- delivery state
- billing, quota, and abuse-control metadata

The relay should not store plaintext task payloads or plaintext private keys.

## Recovery

Recovery should be explicit:

- If the user loses a device, they can revoke it from the account console.
- New devices require account login and device authorization.
- Optional cloud backup can exist, but it should store only ciphertext encrypted
  by a recovery secret, passkey-backed secret, or user-held recovery phrase.
- Without recovery backup, losing the only device can mean losing access to old
  locally encrypted history.

## Current Implementation Mapping

Current terms map to the product model like this:

```text
client_id / relay_token  -> early per-client relay credential
owner_id                 -> early account/owner placeholder
agent handle             -> product agent handle
device_id                -> local client device identity
local API token          -> local daemon access guard
agent keypairs           -> protocol signing/encryption identity
```

The direction is to keep the protocol identity strong while making onboarding
look like a normal account product:

```text
Sign up
Verify email
Create default agent, for example @alice/agent
Authorize this device
Click an invite
Continue as @alice/agent
```
